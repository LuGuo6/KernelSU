# 分析计划：root 权限丢失问题

## 问题描述

- 空配置时 root 权限正常
- 有配置时（无论配置什么文件）root 权限丢失

## 分析结果

### 关键发现

**原始代码（正常）**：
```c
void on_post_fs_data(void)
{
    ksu_load_allow_list();
    ksu_observer_init();
    release_autorun_binary();  // 释放文件到 /data/adb/autorun
    fix_file_context("/data/adb/autorun", "u:object_r:system_file:s0");
    fix_file_permissions("/data/adb/autorun", 0777);  // ← 关键：设置了文件权限
    // ...
}
```

**我的代码（异常）**：
```c
void on_post_fs_data(void)
{
    ksu_load_allow_list();
    ksu_observer_init();
    release_autorun_files();  // 释放文件 + 设置 SELinux 上下文
    // 没有调用 fix_file_permissions！
    // ...
}
```

### 问题根源

**不是 SELinux 上下文的问题，而是文件权限的问题！**

原始代码在释放文件后调用了 `fix_file_permissions("/data/adb/autorun", 0777)` 来设置文件权限为 0777。而我的版本没有调用这个函数。

### 为什么文件权限会影响 root 权限？

1. **文件权限 0755 vs 0777**：
   - 0755：所有者有 rwx，组和其他用户有 rx
   - 0777：所有者、组和其他用户都有 rwx

2. **可能的影响**：
   - 如果某些系统组件需要以其他用户身份访问该文件
   - 如果 SELinux 策略要求文件权限为 0777
   - 如果某些 init 脚本或服务需要以 root 以外的身份执行该文件

3. **连锁反应**：
   - 文件权限不正确 → 某些组件无法正常执行 → 系统状态异常 → root 权限丢失

## 修复方案

### 方案 1：添加 fix_file_permissions 函数

在 `boot_event.c` 中添加 `fix_file_permissions` 函数，并在释放文件后调用：

```c
// Helper: fix file permissions
static void fix_file_permissions(const char *path, umode_t mode)
{
    struct path p;
    int error = kern_path(path, LOOKUP_FOLLOW, &p);
    if (error) {
        pr_err("KernelSU: kern_path failed for %s, err %d\n", path, error);
        return;
    }

    struct inode *inode = p.dentry->d_inode;
    if (!inode) {
        pr_err("KernelSU: no inode for %s\n", path);
        path_put(&p);
        return;
    }

    inode_lock(inode);
    inode->i_mode = (inode->i_mode & S_IFMT) | (mode & 07777);
    mark_inode_dirty(inode);
    inode_unlock(inode);

    pr_info("KernelSU: set permissions %o for %s\n", mode, path);
    path_put(&p);
}

// 在 release_autorun_files 中调用
if (write_file(entry->target_path, entry->start, size, entry->mode) == 0) {
    pr_info("KernelSU: released %s (%zu bytes)\n", entry->target_path, size);
    fix_file_context(entry->target_path, "u:object_r:system_file:s0");
    fix_file_permissions(entry->target_path, 0777);  // ← 添加这一行
}
```

### 方案 2：使用配置文件中的权限

如果希望使用配置文件中指定的权限，可以修改为：

```c
if (write_file(entry->target_path, entry->start, size, entry->mode) == 0) {
    pr_info("KernelSU: released %s (%zu bytes)\n", entry->target_path, size);
    fix_file_context(entry->target_path, "u:object_r:system_file:s0");
    fix_file_permissions(entry->target_path, entry->mode);  // 使用配置的权限
}
```

## 验证步骤

1. 修改 `boot_event.c`，添加 `fix_file_permissions` 函数
2. 在释放文件后调用 `fix_file_permissions(path, 0777)`
3. 重新编译内核模块
4. 测试 root 权限是否恢复正常

## 注意事项

- `fix_file_permissions` 函数需要访问内核内部结构（`inode`），需要适当的内核头文件
- 修改文件权限时需要持有 inode 锁，避免并发问题
- `mark_inode_dirty` 会标记 inode 为脏，需要确保文件系统正确处理
